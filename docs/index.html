Problem

Why Code, Not Labels?

Traditional robot control maps language to fixed action labels — basically classification. It can't handle novel spatial reasoning, composition, or conditionals.

The Old Way

LLM outputs a JSON blob choosing between move_block and stack_block. No reasoning, no spatial computation, no composition. Fails on any instruction outside the predefined set.

Our Approach

The LLM writes executable Python that discovers the scene, computes spatial targets, and sequences actions using loops, conditionals, and helper functions — all from a single natural language instruction.

The Challenge

Generated code must be syntactically valid, safe to execute, and correct — all on the first try, with no scene state in the prompt. The LLM must call get_scene_state() to discover the world.

Method

System Architecture

Two cleanly separated modules: a robosuite simulation backend and an LLM code-generation pipeline.

Key Design Decisions

Type-Dispatch API

pick_and_place(id, target) handles both placement (array) and stacking (int). One primitive, two modes — reduces LLM confusion.

No Scene in Prompt

The LLM receives zero scene state. Generated code must call get_scene_state() to discover blocks, forcing genuinely dynamic programs.

Multi-Layer Safety

AST parsing blocks imports, exec, eval, file I/O, and network calls. Code runs in a restricted namespace with only the allowed API.

Generated Code Samples

Real outputs from the system — no cherry-picking.

Conditional Reasoning SUCCESS 
# "If the red block is to the left of the
# blue block, swap them."

scene = get_scene_state()
red  = next(b for b in scene if b["color"] == "red")
blue = next(b for b in scene if b["color"] == "blue")

if red["pos"][0] < blue["pos"][0]:
    red_pos  = list(red["pos"])
    blue_pos = list(blue["pos"])
    pick_and_place(red["id"],  blue_pos)
    pick_and_place(blue["id"], red_pos)
Multi-Step Composition SUCCESS 
# "Put all the large blocks on the left,
# then stack small blocks on the right."

scene = get_scene_state()
large = [b for b in scene if b["size"] == "large"]
small = [b for b in scene if b["size"] == "small"]

left_pos = make_line_positions(
    get_corner_pos("bottom left"),
    get_corner_pos("top left"), len(large))

for block, pos in zip(large, left_pos):
    pick_and_place(block["id"], pos)

if small:
    right = get_side_pos("right")
    pick_and_place(small[0]["id"], right)
    for b in small[1:]:
        pick_and_place(b["id"], small[0]["id"])
Rejection SUCCESS 
# "Cut the block in half."

say("I cannot cut blocks; the robot can only pick and place objects.")
Discussion

Reflections

What Worked

Code-as-Policies excels at compositional tasks (83.3% on multi-step). Few-shot prompting with just 11 examples taught the LLM to use our custom API idiomatically. Rejection detection was perfect.

Primary Failure Mode

All 7 failures were caused by the LLM generating import numpy statements, which the sandbox blocks. The model defaults to standard Python practices rather than using pre-injected helpers.

Lessons Learned

API design matters more than prompt tuning. The type-dispatch pattern for pick_and_place eliminated an entire class of errors. Good abstractions reduce LLM cognitive load.

Future Work

Retry with error feedback, real robosuite physics testing, multi-model comparison (GPT-4, Claude), and self-verification where the LLM checks the scene after execution.
